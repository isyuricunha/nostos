# Nostos v0.1.0 Release Plan

Last updated: 2026-06-23

## Current-State Audit

The repository is on `main` at `0d1443c2a535c1a9a5894f2566450c82038e2867` with a clean worktree at the start of this release milestone. The remote is `git@github.com:isyuricunha/nostos.git`, and the Go module/import path audit currently shows the canonical owner `github.com/isyuricunha/nostos`.

The current codebase contains a substantial Version 0.1 foundation:

- Go backend with explicit packages for auth, providers, chat, agents, memory, MCP, tasks, replies, feedback, health, database, and worker runtime.
- PostgreSQL and SQLite migrations, repository layers, and integration tests.
- Svelte/Vite frontend with modular views, shared components, design tokens, and a dark command-center shell.
- Dockerfile, Compose files, `.env.example`, Makefile, documentation, and CI workflow.
- Previous validation evidence in `docs/v0.1.0-validation.md`, including Docker/browser smoke work from the frontend redesign milestone.

Previous validation is useful context, but it is not sufficient release proof. This milestone must re-run release-critical checks after fixes are complete.

## Verified Completed Requirements

- Repository identity uses `github.com/isyuricunha/nostos` in Go imports.
- Frontend redesign exists and is modularized under `web/src/views`, `web/src/components`, and `web/src/styles`.
- Authentication, provider, chat, agent, memory, MCP, task, feedback, reply, and settings surfaces exist in the frontend.
- Backend services and repositories exist for all Version 0.1 domains.
- Database migrations exist for both PostgreSQL and SQLite.
- Docker and Compose deployment files exist.
- CI currently covers Go formatting, module tidiness, vet, Go tests, integration tests, frontend lint/check/test/build, Go build, Docker build, and stale module-path guard.

## Release Blockers Found During Audit

These items were verified from current code or previous validation documents at the start of this milestone.

1. Conversation summary regeneration can fail after the API returns `202`.
   - The full path from enqueue to worker execution and UI refresh needs verification.
   - Too-short conversations need a clear bounded result instead of a generic failure.
   - Summary failures need safe, specific errors in task-run events and the conversation summary state.

2. Diagnostics report false `not_configured` states.
   - `internal/health/health.go` hard-codes worker, scheduler, provider, and MCP health as `not_configured`.
   - Worker and scheduler heartbeats are not persisted in a shared status table.
   - Diagnostics need real counts and stale/unknown/degraded distinctions.

3. Empty API collections can serialize as `null`.
   - Frontend defensive `?? []` handling must remain.
   - Backend list responses must consistently encode empty slices as `[]`.

4. Regeneration can expose internal instructions as visible user messages.
   - `internal/chat/service.go` appends `Regeneration instruction:` to the user message content.
   - Regeneration guidance must stay internal to provider context and branch metadata.

5. Conversation titles remain `New conversation`.
   - Existing title recalculation is deterministic and bounded, but it needs release validation and clearer behavior for async recalculation, manual titles, provider failures, and fallback titles.

6. Task-run tool activity is not structured enough.
   - Chat tool calls are persisted, but task-run tool activity is still primarily exposed through human-readable run events.
   - A unified or dedicated task tool-call persistence path is required for queryable task-run details.

7. Permanent browser E2E coverage is missing.
   - Prior browser checks were manual/temporary.
   - A committed Playwright suite using mock provider and mock MCP server is required.

8. Docker deployment needed simplification.
   - Normal external-PostgreSQL deployment should work with `docker compose up -d --build`.
   - At audit time, build metadata used non-release placeholder values outside explicitly injected builds.
   - PostgreSQL 17 must be validated.

9. Release validation is incomplete.
   - Previous validation explicitly listed no release tag, missing GitHub Actions proof, unverified E2E, diagnostics limitations, and task tool-call persistence limitations.

## Implementation Status From This Milestone

- Summary regeneration now returns a clear too-short-history result instead of an opaque failure.
- Runtime diagnostics now use database-persisted worker and scheduler heartbeats plus real provider/MCP counts.
- List endpoints normalize empty collections to JSON arrays.
- Regeneration guidance is passed as internal provider context and is no longer stored as a visible user message.
- Task-run MCP tool activity is persisted as structured task tool calls and returned on task-run details.
- A committed Playwright E2E suite validates the main workspace flow with a mock provider and mock MCP server.
- Build metadata is injected by Makefile, Docker builds, Compose, and CI defaults.
- Compose now passes documented environment values into app and worker containers and validates with PostgreSQL 17 and SQLite.

Remaining release blockers before tagging:

- Push the completed commits.
- Verify the GitHub Actions run for the final commit is green.
- Create and push the annotated `v0.1.0` tag only after CI passes.

## Defects Found During Fresh Audit

- `cmd/app/main.go` defaulted to development version and placeholder build metadata.
- `internal/health/health.go` reports `worker_heartbeat`, `scheduler_heartbeat`, `provider_health`, and `mcp_health` as `not_configured`.
- `internal/chat/service.go` stores regeneration guidance by modifying the visible user message content.
- `docs/v0.1.0-validation.md` records previous runtime output with unknown build metadata and diagnostics gaps.
- CI tested a newer PostgreSQL service, but PostgreSQL 17 support is a release requirement.

## Database Changes Required

- Add a forward migration for runtime component heartbeats/statuses shared by server and worker.
- Add task-run tool-call persistence, either by extending the existing tool-call model safely or by adding a dedicated table.
- Preserve upgrade compatibility from the current migration state for both PostgreSQL and SQLite.
- Add indexes for runtime heartbeats and task-run tool-call lookups.

## API Changes Required

- Normalize list endpoint responses so empty collections encode as `[]`.
- Add or complete diagnostics response fields for server, database, migrations, worker heartbeat, scheduler heartbeat, queue depth, active runs, provider counts, MCP counts, and build metadata.
- Expose structured task-run tool calls where task-run detail APIs can return them.
- Keep regeneration instructions internal while preserving branch behavior and feedback association.
- Improve summary failure/result messages and summary-state refresh behavior.

## Frontend Changes Required

The redesign should be preserved. UI changes should be limited to release correctness:

- Display precise summary errors and too-short summary results.
- Render diagnostics from real structured fields.
- Render task-run structured tool calls when available.
- Keep defensive `?? []` guards while backend responses are fixed.
- Ensure regeneration no longer shows internal instructions in the message list.
- Add Playwright selectors or accessibility labels only where needed for reliable E2E coverage.

## Migration Strategy

- Do not edit already-applied migrations as the only upgrade path.
- Add new numbered PostgreSQL and SQLite migrations.
- Verify both fresh databases and upgrades from the existing current schema.
- Keep portable text status values and UTC timestamps.

## Security Risks To Review

- Provider and MCP secret encryption and write-only replacement semantics.
- Diagnostics must not expose secrets, raw stack traces, connection URLs with credentials, or authorization headers.
- Tool inputs/results must stay bounded and redacted in chat and task contexts.
- MCP stdio execution must avoid shell interpolation and terminate child processes.
- Cookies, CSRF/origin protection, login throttling, and disabled-user session invalidation must be re-verified.
- Container runtime must remain non-root, unprivileged, and without Docker socket mounts.
