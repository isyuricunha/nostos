# Nostos v0.1.0 Completion Plan

## Current-State Audit

The repository is on `main` at `428d0e062b5e1c55512603353b7313411fd3965c`, with `origin` set to `git@github.com:isyuricunha/nostos.git`. The upstream branch is current after `git fetch --prune origin`.

There is no local `AGENTS.md`; the active repository rules are the user-provided rules in this task. The current tree already contains a substantial foundation: Go backend packages, explicit PostgreSQL and SQLite migrations, a Svelte/Vite frontend, Docker and Compose files, CI, docs, and tests. The implementation is not a complete v0.1.0 because several features are represented only by schema, route, minimal UI, or placeholder logic.

The first mandatory correction is repository identity. At the start of this remediation, the module declaration and Go imports used the obsolete GitHub owner. The canonical module and repository path must be `github.com/isyuricunha/nostos`.

## Confirmed Working Behavior

- CLI entrypoint exists with `server`, `worker`, `migrate`, `doctor`, and `version` commands.
- Configuration loading validates required production settings, database driver, timezone, and core worker/chat settings.
- Database migrations are explicit for PostgreSQL and SQLite, with `schema_migrations` tracking.
- SQLite opens with foreign keys, busy timeout, and WAL pragmas.
- Authentication foundation exists: first owner setup, bootstrap owner, bcrypt password hashing, session cookies, CSRF token checks, logout/revoke APIs, login throttling, and audit events.
- Provider CRUD, encrypted direct API keys, `env:` references, model refresh, and streaming OpenAI-compatible chat client exist.
- Conversations, messages, chat runs, cancellation flagging, regeneration, and branch tables exist.
- Agents and explicit memories have CRUD-level services, default agent creation, memory ranking, and run-memory association.
- MCP server CRUD, stdio and HTTP JSON-RPC discovery, tool storage, global per-tool permission, and direct `allow` execution exist.
- Task CRUD, schedules, queued runs, basic claiming, retries, and run events exist.
- Feedback and reply-intent draft services exist with tests.
- The SPA has all top-level navigation entries and can call backend APIs, but it is concentrated in `App.svelte`.
- Docker and Compose files build one image for server and worker, run as non-root for app containers, and support PostgreSQL and SQLite modes.

## Confirmed Incomplete Behavior

- Repository identity is wrong across Go imports and module metadata.
- Chat prompt construction does not include real previous conversation messages. It sends only agent prompt, summary, selected memories, and the latest user message.
- Branch context is not reconstructed from ancestors; branch messages are stored but not used to build provider context.
- Tool-call messages and tool result messages are not reconstructed into provider context.
- Conversation summaries now have metadata, automatic queueing, worker generation, manual regeneration, and clearing APIs. Frontend summary controls remain incomplete.
- The `update_conversation_summaries` system task now dispatches to the chat summarization pipeline.
- Internal maintenance tasks now dispatch to typed handlers and perform bounded database, health-check, chat, summary, event-pruning, lease-recovery, and temporary-file cleanup work. Some deeper release requirements still need hardening, including chat-run heartbeat awareness, generated-title source tracking, scheduler coordination, and broader cross-database integration coverage.
- MCP `ask` tools are not exposed to the model through a complete persisted approval/resume flow.
- Tool calls are not fully persisted with input, output, approval state, timing, error, and truncation data.
- Agent MCP server assignments and agent-specific tool permissions are not enforced during chat.
- Tool execution is a single follow-up pass, not a bounded multi-iteration agent loop.
- Agent runtime fields such as temperature, maximum tool iterations, default tool permission, assigned MCP servers, fallback model, and active-state failures are not fully enforced.
- Agent tasks execute as direct provider prompts and do not use the same agent runtime, memory selection, MCP permissions, or tool policy model as chat.
- Worker concurrency is sequential; `WORKER_CONCURRENCY` currently controls repeated serial claims.
- Task leases are not renewed while long tasks run.
- Scheduler leadership is based only on schedule idempotency, not coordinated database leadership.
- Task concurrency policies are not fully enforced.
- One-time schedules are not disabled after firing.
- Diagnostics contain placeholder component health values.
- Management screens are incomplete editors for agents, memories, MCP, tasks, providers, summaries, approvals, and diagnostics.
- The frontend is too concentrated in `web/src/App.svelte`.
- Documentation currently overstates some behavior and explicitly mentions maintenance placeholders in the README.

- Add any missing task-run fields needed for lease renewal, root retry lineage, cancellation reason, and scheduler coordination.
- Add scheduler coordination storage or deterministic lock rows for both dialects.
- Add worker heartbeat and scheduler heartbeat storage for truthful diagnostics.
- Completed: add provider health latency storage for maintenance and diagnostics.
- Add settings keys for summarization provider/model where needed.

Fresh migrations and upgrades from the current `001_init.sql` state must both work.

## Security Risks

- `ask` tools without persisted approval/resume can encourage unsafe shortcuts; implement approval as durable state.
- Tool-name collisions are possible because global tool names are exposed directly; provider-facing names need stable disambiguation mapped to internal tool IDs.
- Tool output must be bounded, redacted, persisted, and disclosed when truncated.
- MCP stdio execution must never use shell interpolation and must confine inherited environment variables.
- HTTP MCP URLs need validation and timeout limits; secrets must not appear in logs or diagnostics.
- System cleanup tasks must be path-confined and bounded.
- Diagnostics must not expose provider/MCP/database/filesystem raw secrets.
- Authorization and workspace scoping must be verified on every new API path.
