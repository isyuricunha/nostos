## Security Review Notes

- Provider and MCP secrets are encrypted by application code or supplied through `env:` references.
- Provider/MCP secret values are not returned by the API after storage.
- The frontend uses write-only replacement fields for provider API keys, MCP headers, and MCP environment values.
- Secure cookie mode remains configurable with `SECURE_COOKIES`.
- MCP stdio execution remains isolated from shell interpolation in the backend.
- Docker deployment uses non-root distroless runtime, read-only containers, dropped capabilities, no Docker socket, and no privileged mode.

## Known Limitations

These are real remaining limitations from this local remediation pass and should not be described as completed release acceptance:

- GitHub Actions was not inspected after pushing because these commits were not pushed.
- The release tag `v0.1.0` was not created.
- Frontend modularization is improved but `App.svelte` still owns several workflows and remains large.
- Diagnostics still report worker and scheduler heartbeat components as `not_configured`; the database readiness and migration status are now truthful.
- Dedicated task-run tool-call persistence is still represented through task-run events rather than a task-specific tool-call table.
- Scheduler coordination now uses atomic occurrence claiming plus idempotency, not PostgreSQL advisory locks.
- End-to-end browser automation was not run; the repository's Go E2E package and frontend unit tests were run.

No external paid API was used.
